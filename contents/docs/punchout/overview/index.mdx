The PunchOut fromIdentity customer pricing feature enables SAP Ariba buyers to see their negotiated pricing automatically when browsing your Shopify store, without requiring them to log in or create a Shopify account.

## What Is PunchOut?

**PunchOut** is a procurement protocol that allows buyers to "punch out" from their procurement system (like SAP Ariba) to browse a supplier's catalog with their specific pricing, then "punch back" to their procurement system with selected items.

Traditional PunchOut implementations only handle the final cart transfer. This feature extends PunchOut by providing **real-time custom pricing throughout the browsing experience**.

## The fromIdentity Parameter

### What Is fromIdentity?

The `fromIdentity` parameter is a unique identifier sent by SAP Ariba (and other procurement systems) that represents the buyer's identity. This is typically the buyer's SAP Ariba buyer ID.

**Example**:
```
https://yourstore.com/products/widget?fromIdentity=AN01631249352
```

When a buyer clicks a product link in their SAP Ariba catalog, this parameter is automatically included in the URL.

### How It Works

<Step>
  <StepItem title="Buyer Browses SAP Ariba Catalog">
    The procurement user searches for products in their SAP Ariba PunchOut catalog, which lists your store's products.
  </StepItem>
  <StepItem title="Click-Through to Store">
    When they click a product, SAP Ariba opens your store's URL with the `fromIdentity` parameter:
    ```
    yourstore.com/products/item?fromIdentity=AN01631249352
    ```
  </StepItem>
  <StepItem title="Tracker Captures Identity">
    The fromIdentity tracker script (installed in your theme) detects the parameter and stores it in the browser's sessionStorage.
  </StepItem>
  <StepItem title="Widget Resolves Customer">
    The Product Page Widget reads the fromIdentity from sessionStorage and calls the backend API to resolve it to a Shopify customer ID.
  </StepItem>
  <StepItem title="Custom Pricing Displayed">
    The widget fetches and displays the customer's specific pricing, just as if they were logged into their Shopify account.
  </StepItem>
</Step>

## Architecture Components

### 1. Tracker Script (Frontend)

**File**: `fromIdentity-tracker.js`  
**Size**: 1.47 KB (0.66 KB gzipped)  
**Location**: Shopify theme assets

**Responsibilities**:
- Runs on every page load
- Checks URL for `fromIdentity` parameter
- Stores value in sessionStorage with 12-hour TTL
- Provides console logging for debugging

**Example Code Flow**:
```javascript
// On page load
if (urlParams.has('fromIdentity')) {
  const fromIdentity = urlParams.get('fromIdentity');
  sessionStorage.setItem('weevio_fromIdentity', JSON.stringify({
    value: fromIdentity,
    timestamp: Date.now()
  }));
  console.log('[Weevio] Captured fromIdentity:', fromIdentity);
}
```

### 2. Session Manager (Frontend)

**Module**: `fromIdentityManager.ts`  
**Location**: Widget codebase

**Responsibilities**:
- Reads fromIdentity from sessionStorage
- Validates timestamp (12-hour expiration)
- Provides clean API for widget components
- Handles edge cases (expired sessions, invalid data)

**API**:
```typescript
// Get current fromIdentity (if valid)
const fromIdentity = getFromIdentity(); // Returns string | null

// Check if fromIdentity exists and is valid
const isValid = hasValidFromIdentity(); // Returns boolean
```

### 3. Backend Resolution API

**Endpoint**: `GET /api/v1/shopify/b2b/resolve-from-identity`

**Query Parameters**:
- `fromIdentity` (required): SAP Ariba buyer ID
- `domain` (required): Shopify store domain
- `teamID` (optional): Weevio team identifier

**Response (Success)**:
```json
{
  "success": true,
  "data": {
    "customerId": "gid://shopify/Customer/7234567890",
    "priceCode": "IPG_TIER_1"
  }
}
```

**Response (Not Found)**:
```json
{
  "success": false,
  "message": "No customer found with fromIdentity"
}
```

**Caching**: 15 minutes (Cache-Control header)

### 4. Product Page Widget Integration

**Component**: `PriceDisplay.tsx`

**Responsibilities**:
- Checks for fromIdentity in sessionStorage
- Calls resolution API if fromIdentity exists
- Fetches customer-specific pricing
- Displays pricing (or falls back to public pricing)
- Handles errors gracefully

**Pricing Resolution Flow**:
```
1. Widget loads on product page
2. Check sessionStorage for fromIdentity
3. If found: Call /resolve-from-identity API
4. If successful: Fetch customer pricing
5. Display customer pricing
6. If any step fails: Display public pricing
```

## Customer Tagging System

### Tag Format

Shopify customers must be tagged with their SAP Ariba buyer ID. The system recognizes these tag patterns:

**Accepted Prefixes** (case-insensitive, spaces/underscores flexible):
- `Ariba Buyer:`
- `SAP Ariba Buyer:`
- `PunchOut Buyer:`

**Example Tags**:
```
Ariba Buyer: AN01631249352
SAP Ariba Buyer: AN01631249352
PunchOut Buyer: AN01631249352
ariba_buyer:AN01631249352
ARIBA BUYER: AN01631249352
```

**Optional Price Code Tag**:
```
PRICE CODE: IPG_TIER_1
price_code:CORPORATE_DISCOUNT
```

### Tag Resolution Logic

When the backend receives a `fromIdentity` value (e.g., `AN01631249352`), it:

1. **Queries Shopify** for customers with tags matching the patterns above
2. **Extracts** the buyer ID from the tag (part after the colon)
3. **Compares** the extracted ID to the `fromIdentity` parameter
4. **Returns** the matching customer's Shopify ID and price code (if tagged)

**Example Query**:
```graphql
query {
  customers(first: 1, query: "tag:AN01631249352") {
    edges {
      node {
        id
        tags
      }
    }
  }
}
```

## Session Persistence

### Storage Method

The feature uses **sessionStorage** (not cookies or localStorage) for privacy and multi-buyer support:

| Feature | sessionStorage | Cookie | localStorage |
|---------|---------------|--------|--------------|
| **Per-tab isolation** | ✅ Yes | ❌ No | ❌ No |
| **Auto-expiration** | ✅ On tab close | ⚠️ Manual | ❌ No |
| **Privacy-friendly** | ✅ No tracking | ❌ Trackable | ⚠️ Persistent |
| **Multi-buyer support** | ✅ Different tabs | ❌ Shared | ❌ Shared |

### Session Lifecycle

<Step>
  <StepItem title="Session Start">
    Buyer clicks through from SAP Ariba with `?fromIdentity=AN01631249352` parameter.
  </StepItem>
  <StepItem title="Capture & Store">
    Tracker script saves to sessionStorage with current timestamp:
    ```json
    {
      "value": "AN01631249352",
      "timestamp": 1699118400000
    }
    ```
  </StepItem>
  <StepItem title="Active Session (< 12 hours)">
    Widget reads fromIdentity and displays customer pricing on all product pages.
  </StepItem>
  <StepItem title="Session Expiration">
    After 12 hours OR when tab is closed, sessionStorage is cleared and buyer sees public pricing.
  </StepItem>
</Step>

### Multi-Tab Behavior

**Scenario**: Two buyers on the same computer, different browser tabs

**Tab 1**: Opens with `?fromIdentity=BUYER_A`  
→ sessionStorage stores `BUYER_A`  
→ Displays pricing for Customer A

**Tab 2**: Opens with `?fromIdentity=BUYER_B`  
→ sessionStorage stores `BUYER_B` (separate tab storage)  
→ Displays pricing for Customer B

Each tab maintains its own isolated session, enabling multiple buyers to browse simultaneously.

## Caching Strategy

### Two-Level Cache

**Level 1: Backend (15 minutes)**
- Caches customer resolution results
- Reduces Shopify API calls
- Improves response time
- HTTP Cache-Control header

**Level 2: Frontend (12 hours)**
- Stores fromIdentity in sessionStorage
- Eliminates repeated parameter parsing
- Maintains session across page navigation
- Automatic expiration

### Cache Invalidation

**Backend Cache (15 min)**:
- Automatically expires after 15 minutes
- Can be manually cleared via admin tools
- New tags or customer updates reflected within 15 minutes

**Frontend Cache (12 hours)**:
- Automatically expires after 12 hours
- Manually cleared by closing the browser tab
- Buyer can refresh by clicking through from SAP Ariba again

## Security Considerations

### Input Validation

**fromIdentity Parameter**:
- Sanitized to prevent XSS attacks
- Limited to alphanumeric characters and standard symbols
- Maximum length enforced
- Invalid values rejected with error response

**API Endpoint**:
- Rate limiting applied (standard API limits)
- Authentication required for backend access
- HTTPS only (no unencrypted transmission)
- CORS policies enforced

### Privacy Protection

**No Tracking**:
- No cookies set
- No cross-site tracking
- No personally identifiable information logged
- sessionStorage cleared on tab close

**Data Minimization**:
- Only stores buyer ID (no personal data)
- No browsing history tracked
- No analytics collection specific to PunchOut users

## Performance Impact

### Bundle Size
- **Tracker Script**: 1.47 KB (0.66 KB gzipped)
- **Session Manager**: ~2 KB (included in widget)
- **Widget Integration**: ~1 KB additional code
- **Total Impact**: ~4.5 KB (minimal)

### Runtime Performance
- Tracker runs once per page load (~1ms)
- Session check is synchronous read from sessionStorage (~0.1ms)
- API call cached on backend (15 min TTL)
- No impact on page load time or rendering

### Scalability
- Backend resolution API can handle 1000+ requests/second
- Caching reduces Shopify API load by 99%
- No database bottlenecks (Shopify handles queries)

## Graceful Degradation

The feature is designed to fail gracefully:

| Failure Scenario | Behavior |
|-----------------|----------|
| Tracker script fails to load | Widget shows public pricing (standard behavior) |
| sessionStorage unavailable | Widget shows public pricing |
| Resolution API timeout | Widget shows public pricing |
| Customer not found | Widget shows public pricing |
| Invalid fromIdentity format | Widget shows public pricing |
| Expired session (> 12 hours) | Widget shows public pricing |

**Result**: Buyers always see pricing (public or custom), never a blank page or error message.

## Comparison with Traditional PunchOut

### Traditional PunchOut (Cart Transfer Only)

**What it does**:
- Buyer browses catalog with **public pricing**
- Adds items to cart
- Clicks "Transfer Cart" to punch back to SAP Ariba
- **SAP Ariba applies custom pricing** when importing cart

**Limitations**:
- Buyer doesn't see their real prices while browsing
- Can cause confusion ("Why does SAP show different prices?")
- Requires buyer to add items to cart to see custom pricing

### Enhanced PunchOut (fromIdentity Feature)

**What it does**:
- Buyer browses catalog with **custom pricing from the start**
- Sees accurate totals before adding to cart
- Cart transfer still works normally
- **Consistent pricing** in store and SAP Ariba

**Benefits**:
- Better buyer experience
- Reduces "cart abandonment" due to price confusion
- More accurate purchasing decisions
- Maintains cart transfer compatibility

## Related Features

This feature complements the existing PunchOut cart transfer functionality. Both features can be used together:

- **fromIdentity pricing** (this feature): Displays custom pricing while browsing
- **Cart transfer**: Sends cart contents back to SAP Ariba for purchase

<Note type="info">
The fromIdentity feature does NOT replace cart transfer. It enhances the browsing experience while maintaining all existing PunchOut capabilities.
</Note>
